## Key Architectural Decisions

### 1. Why OpenSpec Over Spec Kit?

| Criterion | OpenSpec | Spec Kit |
|-----------|----------|----------|
| Output Size | 250 lines | 800 lines |
| Learning Curve | Lower | Higher |
| Flexibility | High | Medium |
| Documentation Burden | Lightweight | Heavy |
| Best For | 0→1 and 1→n changes | Large platform teams |

**Decision**: Use OpenSpec for lightweight, focused specifications.

### 2. Why NATS Over Redis?

| Feature | NATS JetStream | Redis |
|---------|----------------|-------|
| Message Delivery | At-least-once, exactly-once ✅ | At-most-once ❌ |
| Persistence | Built-in ✅ | Requires AOF/RDB ⚠️ |
| Work Queues | Pull consumers ✅ | Lua scripts ⚠️ |
| Clustering | Native ✅ | Complex sharding ⚠️ |
| DAG Coordination | Perfect fit ✅ | Workarounds needed ❌ |
| Operational Complexity | Lower ✅ | Higher ⚠️ |

**Decision**: Use NATS for all distributed coordination.

### 3. Why Hybrid Verification?

| Method | Speed | Accuracy | Coverage |
|--------|-------|----------|----------|
| NL Assertions (IntentGuard) | Fast (5s) | 85-92% | Broad |
| Property-Based (Hypothesis) | Medium (30s) | 95-98% | Edge cases |
| Symbolic (Z3) | Slow (60s) | 99.9% | Critical paths |

**Decision**: Use all three in layers - trust hierarchy: Symbolic > Property > NL.

### 4. Why Agile-Waterfall Hybrid?

**Waterfall alone**:
- ❌ No working software until end
- ❌ Can't adapt to changes
- ❌ Late feedback

**Agile alone**:
- ❌ Insufficient upfront design
- ❌ Architecture drift
- ❌ No formal review gates

**Hybrid**:
- ✅ Comprehensive architecture upfront
- ✅ Working software every 2 weeks
- ✅ Formal gates prevent moving forward with issues
- ✅ Agile adaptation within phases
